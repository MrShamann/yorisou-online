## 2. Scored runtime boundary (§7) — NOT a generalized engine

Coverage: 48/48 required → 0..47 = `insufficient_coverage` (no primary/secondary/Mixed, no imputation, exact remaining). Tie-break: declaration order `[anshin, pace, tsunagari, seicho, yakuwari, totonoi, jikkan]`, secondary-ordering only. **Internal win rates live only inside the runtime envelope's `internal` field and are stripped by `assembleYorisouValuesResult` before any public/private payload — never in UI, share, URLs, HTML attrs, logs, analytics, API responses or export.** No universal score; no cross-method numeric comparison.

## 10. YV-1.1 corrective (provenance, confirmation, anonymous completion, test integrity)
Applied on the same draft PR #119 by amending migration `202607210001` IN PLACE (never hosted/merged). No canonical JSON, question/dimension/scoring/result copy, or the `< 0.05` Mixed rule changed.

- **YV-C1 confirmation ≠ correction.** Confirmation is now a distinct mutation. New RPC `yorisou_values_assessment_set_confirmation` updates only the confirmation field — NO version increment, NO version row, NO `corrected` event — and emits one `confirmation_changed` event with reason `user_confirmed` / `user_not_quite` / `user_skipped`. `yorisou_values_assessment_correct` no longer writes confirmation and REJECTS a byte-equivalent (no-op) answer change (`values_no_answer_change`), so a version is created only on a genuine answer change. The API PATCH accepts exactly one operation (answers XOR confirmation).
- **YV-C2 stored-record provenance enforced.** Both mutating RPCs re-verify the full 6-field provenance against the locked row inside the same transaction (no TOCTOU gap) and raise `values_record_contract_version_mismatch` on any drift. The API GET/PATCH re-check the stored record against canonical (`recordProvenanceMatchesCanonical`) and fail closed (409) rather than reinterpreting a stale record under changed content. A stale record stays deletable (DELETE has no provenance gate).
- **YV-C3 anonymous, non-persistent scoring.** New `POST /api/tests/yorisou-values/score` — same route gate, NO auth, NO DB read/write, answers never logged, returns the canonical result (no internal numerics) with `saved:false`. Persistence remains a separate authenticated `POST /assessments`. The client flow now lets a signed-out visitor see their result and then explicitly sign in to save (device-local pending progress + `/login?next=…`).
- **YV-C4 strict API contracts.** create/score/PATCH reject unknown top-level fields; PATCH also rejects empty and ambiguous (answers+confirmation) bodies; body-size limit and bounded error codes retained.
- **YV-C5 test integrity.** The three vacuous scoring tests were replaced with real proofs (concrete real-bank VAL_R_MIXED fixture; exact 0.05 boundary via a synthetic 1/100-grid bank; genuine side-swapped-bank A/B inversion), plus provenance/confirmation/strict-field/anonymous coverage at DB, contract, and full-stack levels. See `YV1_TEST_INVENTORY.md`.

Pure contract helpers (`CANONICAL_VALUES_PROVENANCE`, `recordProvenanceMatchesCanonical`, `firstUnknownKey`) live in `lib/yorisou/methods/yorisou-values/contract.ts` (no `server-only`) and are re-exported by `lib/server/yorisouValuesApi.ts`. Registry maturity unchanged (`implementation: in_progress`, activation `gated`). No hosted apply, no flag enabled, PR #119 remains OPEN/draft/DO-NOT-MERGE.
