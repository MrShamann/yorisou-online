## 10. YV-1.1 corrective (provenance, confirmation, anonymous completion, test integrity)
Applied on the same draft PR #119 by amending migration `202607210001` IN PLACE (never hosted/merged). No canonical JSON, question/dimension/scoring/result copy, or the `< 0.05` Mixed rule changed.

- **YV-C1 confirmation ≠ correction.** Confirmation is now a distinct mutation. New RPC `yorisou_values_assessment_set_confirmation` updates only the confirmation field — NO version increment, NO version row, NO `corrected` event — and emits one `confirmation_changed` event with reason `user_confirmed` / `user_not_quite` / `user_skipped`. `yorisou_values_assessment_correct` no longer writes confirmation and REJECTS a byte-equivalent (no-op) answer change (`values_no_answer_change`), so a version is created only on a genuine answer change. The API PATCH accepts exactly one operation (answers XOR confirmation).
- **YV-C2 stored-record provenance enforced.** Both mutating RPCs re-verify provenance against the locked row inside the same transaction (no TOCTOU gap) and raise `values_record_contract_version_mismatch` on any drift. The API GET/PATCH re-check the stored record against canonical (`recordProvenanceMatchesCanonical`) and fail closed (409) rather than reinterpreting a stale record under changed content. A stale record stays deletable (DELETE has no provenance gate). **Correction (see §11 / YV-1.2):** YV-1.1 verified only FIVE fields (method/bank/scoring/result-schema versions + bank hash); `method_id` was not explicitly checked. YV-1.2 completes the sixth field.
- **YV-C3 anonymous, non-persistent scoring.** New `POST /api/tests/yorisou-values/score` — same route gate, NO auth, NO DB read/write, answers never logged, returns the canonical result (no internal numerics) with `saved:false`. Persistence remains a separate authenticated `POST /assessments`. The client flow now lets a signed-out visitor see their result and then explicitly sign in to save (device-local pending progress + `/login?next=…`).
- **YV-C4 strict API contracts.** create/score/PATCH reject unknown top-level fields; PATCH also rejects empty and ambiguous (answers+confirmation) bodies; body-size limit and bounded error codes retained.
- **YV-C5 test integrity.** The three vacuous scoring tests were replaced with real proofs (concrete real-bank VAL_R_MIXED fixture; exact 0.05 boundary via a synthetic 1/100-grid bank; genuine side-swapped-bank A/B inversion), plus provenance/confirmation/strict-field/anonymous coverage at DB, contract, and full-stack levels. See `YV1_TEST_INVENTORY.md`.

Pure contract helpers (`CANONICAL_VALUES_PROVENANCE`, `recordProvenanceMatchesCanonical`, `firstUnknownKey`) live in `lib/yorisou/methods/yorisou-values/contract.ts` (no `server-only`) and are re-exported by `lib/server/yorisouValuesApi.ts`. Registry maturity unchanged (`implementation: in_progress`, activation `gated`). No hosted apply, no flag enabled, PR #119 remains OPEN/draft/DO-NOT-MERGE.

## 11. YV-1.2 final corrective (six-field provenance, login continuation, truthful/idempotent confirmation)
Applied on the same draft PR #119 by amending migration `202607210001` IN PLACE (never hosted/merged). No canonical JSON, questions, dimensions, scoring, result IDs/copy, the `< 0.05` Mixed rule, registry activation state, route identity, persistence ownership, or deletion architecture changed. - **YV-C6 — all SIX provenance fields (method_id included).** `CANONICAL_VALUES_PROVENANCE` now carries `methodId`; `recordProvenanceMatchesCanonical` requires exact equality on `method_id, method_version, bank_version, scoring_version, result_schema_version, bank_content_hash`. Both mutation RPCs (`yorisou_values_assessment_correct`, `yorisou_values_assessment_set_confirmation`) take an explicit `p_expected_method_id` and verify all six against the locked row. The table's `method_id` CHECK constraint remains, but is NOT treated as a substitute for the explicit six-field mutation contract. Contract + DB-harness tests drift each of the six fields independently.
- **YV-C7 — complete anonymous→authenticated explicit-save continuation.** Resume now PEEKS pending progress (`readPendingValuesProgress`) instead of consuming it on read. When the pending contract carries all 48 answers the flow opens DIRECTLY on the completed review/save state (the user is never sent back to answer question 48) and nothing is persisted automatically; an authenticated visitor sees an explicit action (「この結果を非公開で保存する」) that creates exactly ONE owned assessment. Pending is cleared ONLY on explicit discard, a successful authenticated save, or a stale/expired entry (which shows the truthful incompatibility notice). A failed save keeps the compatible pending recoverable; the save button guards against duplicate submits so a refresh cannot create a second assessment.
- **YV-C8 — truthful + idempotent confirmation.** The client sets a bounded saving state, updates the displayed confirmation ONLY when the PATCH returns `ok`, shows truthful Japanese error copy on failure, and ignores concurrent clicks while a request is pending. The `set_confirmation` RPC is idempotent: requesting the confirmation the record already holds is a truthful no-op success — NO additional `confirmation_changed` event, `updated_at` NOT bumped, no version change; only a genuine state change emits exactly one content-free event.
